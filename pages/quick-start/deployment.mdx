# Deployment Guide

All Mind Measure projects deploy automatically via Vercel's GitHub integration. Pushing to `main` triggers a build and deploy.

**Never use `npx vercel --prod` or the Vercel CLI.** All deployments must go through git.

**Last Updated**: 28 February 2026

---

## Deployment Method

```
Local changes → git add -A → git commit → git push → Vercel auto-deploy
```

Every project has a GitHub repo connected to a Vercel project. Pushing to `main` triggers:
1. Vercel pulls the latest code from GitHub
2. Runs `npm run build`
3. Deploys to production

**Build times**: ~2–4 minutes for admin dashboards, ~1 minute for docs/hubs.

## Environment Variables

Environment variables are set in each project's Vercel dashboard under Settings → Environment Variables.

**Important**: Variables prefixed with `VITE_` are baked into the client bundle at build time. Never use `VITE_` for secrets that should remain server-side. Adding or changing a `VITE_` variable requires a redeploy to take effect.

Server-side variables (used in `api/` functions) are available at runtime and do not require the `VITE_` prefix.

### Monitoring Variables (core repos only)

| Variable | Required | Description |
|----------|----------|-------------|
| `CRON_SECRET` | Yes | Shared secret for Vercel Cron authentication. Set the same value in Vercel project settings. |
| `ALERT_EMAIL` | Yes | Email address for automated health check alerts. |

These must be set in both `mind-measure-core` and `mind-measure-pro-core` Vercel projects for automated monitoring to work.

## Pre-Deployment Checklist

- [ ] All changes committed (no unstaged files in `git status`)
- [ ] Commit includes ALL modified files (partial commits cause reverts)
- [ ] Environment variables configured in Vercel if new ones were added
- [ ] `VITE_` env vars: trigger a redeploy after adding (build-time only)

## Post-Deployment Verification

- [ ] Application loads without errors
- [ ] Ecosystem health audit passes (`/api/system/ecosystem-audit`)
- [ ] Auth flows working (superuser login, Cognito)
- [ ] Voice agent connects (if mobile app changes)

## Auth-Protected Sites

The documentation site and marketing CMS are protected by Next.js middleware. Direct URL access is blocked. Access is granted via token passthrough from the superuser portals.

The token is set as `VITE_MM_INTERNAL_TOKEN` (client-side, in admin dashboards) and `MM_INTERNAL_TOKEN` (server-side, in docs and marketing CMS). All four must have the same value.
