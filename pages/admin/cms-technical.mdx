# CMS Technical Documentation

Technical documentation for the Mind Measure content management systems — covering the university/company admin CMS (within the admin portals) and the Marketing CMS.

**Last Updated**: 28 February 2026

---

## Architecture Overview

Mind Measure has two CMS contexts:

| CMS | Location | Purpose |
|---|---|---|
| **Institutional CMS** | Built into admin portals (`mind-measure-core`, `mind-measure-pro-core`) | University/company onboarding, settings, users, resources |
| **Marketing CMS** | Standalone app (`mind-measure-marketing-cms`) at [marketing.mindmeasure.co.uk](https://marketing.mindmeasure.co.uk) | Blog posts, content pool, page management |
| **Content Hubs** | Standalone apps (`mind-measure-university-hub`, `mind-measure-pro-hub`) | Article editing, nudge management, content pool browsing |

### Technology Stack

**Institutional CMS (Admin Portals)**
- React 18 + TypeScript + Vite
- Tailwind CSS for styling
- Vercel serverless API functions (`/api/*`)
- AWS Aurora Serverless v2 (PostgreSQL)
- AWS Cognito + custom JWT authentication

**Marketing CMS**
- Next.js 16 + TypeScript
- Prisma ORM + PostgreSQL
- Tiptap rich-text editor
- AWS S3 for media storage
- Auth middleware (token-based, access via superuser portal)

**Content Hubs**
- React + Vite + TypeScript
- Tiptap rich-text editor (same as Marketing CMS)
- Auth passthrough from institutional dashboard

## Authentication

### Institutional CMS (Admin Portals)

Two authentication methods:

**1. Superuser Login** (Mind Measure staff and other allow-listed users)
- Passwordless 6-digit code via email
- `POST /api/auth/send-access-code` → looks up email in `core_users`, generates code, sends via AWS SES
- `POST /api/auth/verify-access-code` → validates code, returns JWT carrying the user's `permissions` array
- Allow-list is the `core_users` table (any active user with at least one permission, regardless of email domain)
- Access to the superuser dashboard requires `super_admin` in `permissions`

**2. University/Company Admin Login**
- Email + invite token flow
- Admin receives invite email with token
- First login sets up account, subsequent logins use token verification
- University-scoped access (can only see their own institution's data)

### Marketing CMS

Protected by Next.js middleware. No standalone login — access is granted via auth token passthrough from the superuser portal tabs.

## Data Access Patterns

### Serverless Function Pattern

All database access goes through Vercel serverless functions in the `/api/` directory. The client-side app calls these endpoints with auth headers.

```typescript
// Client-side call
const response = await fetch('/api/database/select', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({
    table: 'universities',
    filters: { slug: universitySlug }
  })
});
```

```typescript
// Server-side (Vercel function)
import { Client } from 'pg';

export default async function handler(req, res) {
  const userId = await getUserIdFromToken(req.headers.authorization);
  const client = new Client(getSecureDbConfig());

  try {
    await client.connect();
    const result = await client.query(
      'SELECT * FROM universities WHERE slug = $1',
      [req.body.filters.slug]
    );
    res.json({ data: result.rows });
  } finally {
    await client.end();
  }
}
```

**Rules:**
- Use `Client` (not `Pool`) in serverless functions
- Always call `client.end()` in a `finally` block
- Never hardcode credentials — use `getSecureDbConfig()`
- Always verify auth token before processing requests

## University Onboarding (7-Step Wizard)

The institutional CMS includes a university onboarding wizard:

| Step | Component | Data |
|---|---|---|
| 1 | Basic Information | Name, slug, website, contact details |
| 2 | Demographics | Student counts by type (UG, PG, international, etc.) |
| 3 | Academic Structure | Faculties → Schools → Departments hierarchy |
| 4 | Emergency Services | Crisis contacts, mental health services, local resources |
| 5 | Wellbeing Content | Content categories and initial tips |
| 6 | Reports & Analytics | Report configuration, recipients, frequency |
| 7 | Authorised Users | Admin accounts with email invite flow |

Each step saves independently to the database. The wizard can be resumed from any step.

## Marketing CMS (Content Studio)

### Stack
- **Framework**: Next.js 16 (App Router)
- **ORM**: Prisma
- **Editor**: Tiptap rich-text editor with image support
- **Storage**: AWS S3 for media uploads
- **API**: Next.js API routes (`/app/api/`)

### Features
- Blog post management (draft, published, archived)
- Media library with S3 upload
- Content targeting (`targetSites`: university, student, pro)
- Public API for marketing site Research Hub
- Content pool for university/company hubs to browse and adopt

### Content Pool Flow

```
Marketing CMS → Blog Post (targetSites: [university, pro])
                    ↓
            Public API serves to:
            ├── Marketing Website Research Hub
            ├── University Content Hub (pool browser)
            └── Pro Content Hub (pool browser)
                    ↓
            Hub admin "adopts" article → appears in mobile app
```

## Environment Variables

### Admin Portals (Vite — VITE_ prefix required for client-side)

```bash
VITE_BACKEND_PROVIDER=aurora-serverless
VITE_AWS_REGION=eu-west-2
VITE_AWS_COGNITO_USER_POOL_ID=...
VITE_AWS_COGNITO_CLIENT_ID=...
PLATFORM_HANDOFF_SECRET=...  # Server-side: signs the docs/CMS SSO handoff (never VITE_-prefixed)
```

The docs / marketing CMS handoff no longer uses a client-side token. The
backend mints a short-lived signed JWT (`/api/auth/platform-handoff`); the
secret stays server-side, so nothing sensitive is inlined into the client bundle.

Server-side env vars (no VITE_ prefix) are used in `/api/` functions:

```bash
RDS_HOSTNAME=...
RDS_PORT=5432
RDS_DB_NAME=mindmeasure
RDS_USERNAME=...
RDS_PASSWORD=...
```

### Marketing CMS (Next.js — no VITE_ prefix needed)

```bash
DATABASE_URL=...  # Prisma connection string
PLATFORM_HANDOFF_SECRET=...  # Verifies the signed SSO handoff (?mm_sso=) in middleware
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
AWS_REGION=eu-west-2
S3_BUCKET_NAME=...
```
