### Weekly Reflection — Production Wiring

**Priority**: High
**Status**: Phase 1 complete — production wiring in progress

**Completed (Phase 1)**:

- Web view (HMAC-signed URL), PDF via print CSS, share card
- Safeguarding infrastructure — keyword scan, Bedrock safety flag, score threshold hold, audit trail, prompt versioning
- Documentation — feature docs, safeguarding policy, operations runbook

**Remaining work**:
- Welfare-modified reflection for negative-only weeks (Tier 2 — student-facing only, no external sign-off required)
- Opt-out toggle in mobile app settings
- Tier 0 nudge — gentle email/push when student hasn't checked in for 7+ days
- Superuser audit panel — Wellbeing Sends view for Mind Measure staff
- Jodie prompt improvements (see below)
- Pro parity — same pipeline in `mind-measure-pro-core` once university is validated

### Welfare Monitoring System

**Priority**: High
**Status**: Foundation built — Tiers 2/3 escalation pending clinical and legal sign-off

A tiered response framework that sits beneath the Weekly Reflection and acts throughout the week, not just on Sundays.

| Tier | Signal | Action |
|---|---|---|
| 0 | No check-ins for 7+ days | Gentle nudge to student only |
| 1 | Single score < 25 or crisis keywords | In-app message + crisis resources; internal Mind Measure log |
| 2 | Pattern: 3+ check-ins all < 40 | Mid-week welfare nudge + modified Sunday reflection |
| 3 | Week avg < 25 or explicit crisis language | Reflection held; university welfare contact notified (GDPR vital interests) |

**Foundation already built**: keyword scan, Bedrock safety flag, score thresholds, `welfare_nudges_sent` table, full audit trail in `weekly_reflection_sends`.

**Pending sign-off before Tier 2/3 automation**: clinical review of score thresholds, legal review of DPA terms and vital interests basis for Tier 3 disclosure. See [Safeguarding Policy](/operations/weekly-reflection-safeguards).

### Jodie Prompt Improvements

**Priority**: Medium
**Status**: Planned — building after welfare reflection is stable

The quality of the Weekly Reflection depends directly on what Jodie surfaces in conversation. Proposed additions to `jodie_prompt_v2_uni.md`:

- **Named events** — "What's one specific thing that happened this week?" — gets dateable, referenceable moments
- **Sleep/energy probe** — if not raised naturally — the single most predictive wellbeing signal
- **Positive anchor** — before closing, always ask for one good moment — feeds the wins data in the reflection
- **Social connection probe** — "Have you seen anyone this week you're glad you did?" — strong predictor, currently underrepresented
- **Week framing** — "Thinking back across the whole week..." — gets weekly perspective rather than just today

These are additive and do not change the check-in structure. Changes require prompt version bump and regression testing.

### Cohort Template Library

**Priority**: High
**Status**: Active planning — sprint candidate

Evolve the existing Cohort Analysis from a build-it-yourself tool into a shared library of reusable, live-data cohort views. Users build cohort configurations once, and they become discoverable templates that anyone at the institution can use, always populated with current data.

**Current state**: Cohort comparisons are saved to browser `localStorage` with auto-generated titles, no user ownership, no sharing, and placeholder data generation instead of live queries.

**Target state**:
- Saved cohort configurations stored in the database, scoped to institution and user
- Custom titles and author attribution on every saved view
- "Publish to library" toggle makes a user's view available to all colleagues at the same institution
- Browsable library of published templates, searchable by title, author, and cohort type
- Templates populated with live data on load (not static snapshots)
- Historical data overlay: show the same cohort definition over a previous period for comparison (e.g. same term last year)
- Mind Measure can seed institutions with recommended starter templates based on their structure

**Key milestones**:
- Phase 1: Custom chart titles (editable before save) and migration from `localStorage` to database (`saved_cohort_templates` table with `university_id`, `created_by`, `title`, `is_published`)
- Phase 2: Live data population — cohort charts query real data via `UniversityDataService` instead of placeholder generators
- Phase 3: Library UI — browse, search, and load published templates; author attribution
- Phase 4: Historical overlay — query archived data for the same cohort definition across different time periods
- Phase 5: Institutional starter packs — Mind Measure curates recommended templates per institution type

**Why this matters**: A single wellbeing researcher's insight (e.g. comparing halls of residence across campuses) becomes institutional knowledge. The library effect means the value of the platform compounds as more people use it, and researchers spend time interpreting data rather than constructing views.

### Streak Rewards

**Priority**: Under consideration — approach being reviewed
**Status**: Concept — not proceeding until reward mechanism is resolved

The original concept was to reward consistent check-in habits with digital vouchers. This is being reconsidered in light of the Weekly Reflection.

**The tension**: The Weekly Reflection is itself a form of intrinsic reward — recognition, insight, feeling seen. Research on motivation consistently shows that introducing external (extrinsic) rewards for behaviours that are already intrinsically motivated can undermine long-term engagement. Students who check in for a voucher are not students who are building a genuine wellbeing habit.

**Open questions before proceeding**:
- Can a reward mechanism reinforce intrinsic motivation rather than replace it? (e.g. rewards tied to milestones like "your 30th reflection" rather than raw check-in count)
- Is there a model where social recognition (sharing a streak, peer acknowledgement) achieves the engagement goal without the extrinsic reward risk?
- What does the data show — do students who check in regularly already not need an incentive, and does the incentive only attract low-quality engagement?

**Not parked** — the revenue case (partner retailer sponsorship) remains valid. But the mechanism needs design thinking, not just engineering. Revisit after the Weekly Reflection is fully live and engagement data is available.

### Auth Token Standardisation

**Priority**: Medium
**Status**: Planning

The University mobile app (`mind-measure-mobile-final`) and Pro mobile app (`mind-measure-pro-mobile`) use different token retrieval patterns — `setTokenProvider` vs `cognitoApiClient`. Aligning both on the same pattern improves reliability and enables shared API middleware.

**Key milestones**:
- Phase 1: Audit current auth patterns across both mobile apps
- Phase 2: Align on `cognitoApiClient` pattern (proven in Pro)
- Phase 3: Extract shared auth module (feeds into monorepo Phase 2)

### Voice Provider Abstraction & PersonaPlex

**Priority**: Low (R&D)
**Status**: Research — platform audit documents alternatives

Build a provider-agnostic voice interface so ElevenLabs can be swapped for alternatives (OpenAI Realtime, Cartesia Sonic-3, or self-hosted NVIDIA PersonaPlex). PersonaPlex offers 100x cost reduction at scale (~$0.001/min vs ~$0.10/min) with MIT-licensed weights.

**Key milestones**:
- Phase 1: Build interface-based voice provider abstraction layer
- Phase 2: Prototype PersonaPlex on cloud A100 (AWS g5.2xlarge) with wellbeing check-in prompts
- Phase 3: Benchmark quality, latency, and cost vs ElevenLabs
- Phase 4: If viable, migrate to self-hosted; break-even at ~10,000 minutes/month

### Score Reveal Interaction

**Priority**: Medium
**Status**: Spec written — not started

Strengthen the perceived significance of the Mind Measure score by introducing a short transition between processing completion and score display. A score represents an evaluation of the user's state — presenting it instantly can make the result feel trivial or arbitrary. A brief transition creates a natural pause, signalling that the system has completed analysis and is preparing the result.

**Proposed interaction sequence**:
1. User completes check-in
2. Processing animation runs (existing behaviour)
3. Short transition screen appears: "Your check-in is complete. Preparing your results."
4. Score reveal screen loads — gauge appears first, score animates into place over ~1 second, scoring band highlights
5. One-line interpretation appears under the score
6. Full dashboard view becomes available

**Design principles**:
- Keep the transition short (1–2 seconds)
- Avoid technical language or process descriptions
- Emphasise completion and preparation rather than analysis
- Ensure animation timing feels intentional rather than delaying the user

**Priority**: High — foundation built, Tiers 2/3 pending sign-off

The tiered welfare monitoring framework is now partially implemented (see [Welfare Monitoring System](#welfare-monitoring-system) in Active Planning above). **Still to build** (requires clinical and legal sign-off):
