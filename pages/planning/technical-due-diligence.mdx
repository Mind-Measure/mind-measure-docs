# Technical Due Diligence Report

**Platform**: Mind Measure — AI-powered wellbeing monitoring for universities and corporate organisations  
**Date**: 1 March 2026  
**Scope**: Full technical audit across all production systems  

---

## Executive Summary

Mind Measure is a production platform comprising 7 web applications, 6 AWS Lambda functions, and a shared AWS backend. The platform serves two product lines — **University** (student wellbeing) and **Pro** (corporate employee wellbeing) — sharing common infrastructure with separate branding and data isolation.

The technical foundation is enterprise-grade: AWS managed services (Aurora PostgreSQL, Cognito, S3, Bedrock, Rekognition), HIPAA-aligned security architecture, GDPR-compliant data handling, and a consistent React/TypeScript/Vite stack across all applications. The platform includes automated testing (750+ unit tests with enforced coverage thresholds, Playwright e2e), CI/CD pipelines (GitHub Actions), pre-commit quality gates, automated health monitoring with alerting, and Sentry error tracking across all applications.

Code duplication across University and Pro platforms has been eliminated through 4 private shared NPM packages (`@mindmeasure/api-lib`, `shared-services`, `shared-types`, `shared-ui`), consolidating ~15,800 lines into single sources of truth while maintaining repository independence for M&A flexibility.

The core intellectual property — a multimodal fusion engine combining text, audio, and visual analysis — is technically differentiated and non-trivial to replicate.

## 2. Intellectual Property: Multimodal Fusion Engine

The platform's core differentiator is a **3-modality wellbeing assessment engine** that analyses check-in conversations across:

- **Text analysis (70% weight)**: AWS Bedrock (Claude 3.7 Sonnet) extracts mood scores, PHQ-2/GAD-2 clinical indicators, risk markers, sentiment, key themes, positives, and negatives from conversation transcripts.
- **Audio analysis (15% weight)**: Client-side Web Audio API extracts speech patterns — energy levels, speaking rate, vocal stress indicators, silence ratios.
- **Visual analysis (15% weight)**: AWS Rekognition analyses facial expressions during check-ins — emotion classification, engagement indicators.

The fusion engine includes **dynamic fallback**: if one modality fails (e.g., camera permission denied), weights automatically redistribute across available modalities. The system degrades gracefully rather than failing.

Session processing is **asynchronous**: completed check-ins are queued via SQS and processed by Lambda workers, preventing user-facing latency from downstream processing.

This architecture is non-trivial to replicate. The combination of real-time voice AI, multimodal fusion with fallback, and clinical indicator extraction (PHQ-2, GAD-2) in a single check-in flow represents genuine technical IP.

## 3. Security Architecture

### 3.1 Authentication

**Primary authentication**: AWS Cognito with RS256 JWT tokens. All API routes verify JWT signatures against the Cognito JWKS endpoint — tokens are cryptographically verified, not merely decoded.

**Superuser authentication**: Separate flow using time-limited 6-digit codes sent to verified @mindmeasure.co.uk email addresses. Codes are HMAC-hashed (not stored in plaintext). Challenge tokens expire after use.

**MFA**: AWS Cognito MFA with TOTP (Google Authenticator) and SMS support. MFA setup, challenge, and verification endpoints exist at `/api/auth/mfa-setup`, `/api/auth/mfa-challenge`, `/api/auth/mfa-verify`.

### 3.2 Encryption

| Layer | Method | Standard |
|-------|--------|----------|
| Data at rest (database) | AES-256 | AWS KMS, annual key rotation |
| Data at rest (Cognito) | AES-256 | AWS KMS |
| Data in transit (browser → Vercel) | TLS 1.3 | HTTPS enforced |
| Data in transit (Vercel → Aurora) | TLS 1.2+ | SSL required (`sslmode=require`) |
| Data in transit (email) | TLS 1.2 | STARTTLS via SES |
| Field-level PHI encryption | AES-256-GCM | Client-side, authenticated encryption with AAD |

**PHI field-level encryption** (`PHIEncryption.ts`) encrypts sensitive fields individually: first name, last name, email, phone number, date of birth, medical record numbers, conversation transcripts, assessment results, medical notes, diagnostic data. This is defence-in-depth — even if the database is compromised, PHI fields remain encrypted.

### 3.3 Access Control (RBAC)

| Role | Scope | Capabilities |
|------|-------|-------------|
| Student / Employee | Own data only | Check-ins, view own history, export own data |
| University Admin / Company Admin | Scoped to institution | View aggregated analytics, manage users, generate reports |
| Superuser | Platform-wide | All admin functions, system configuration, ecosystem audit |

RBAC is enforced via `requireRole()` middleware on API routes. Data scoping ensures institution admins can only access data within their organisation.

### 3.4 Input Validation & Injection Prevention

- **Parameterised SQL**: Every database query uses `$1, $2` parameterisation — zero SQL injection surface.
- **Table allowlists**: Database endpoints only accept queries against pre-approved tables.
- **Identifier validation**: Column and table names validated against `/^[a-zA-Z_][a-zA-Z0-9_]*$/`.
- **Frame array caps**: Visual analysis endpoints cap input arrays to prevent abuse.
- **Transcript length limits**: Text analysis endpoints enforce maximum input sizes.

### 3.5 CORS Configuration

CORS is **whitelist-based** (not wildcard). Each project's `api/_lib/cors-config.ts` defines an explicit origin allowlist:
- Production domains (admin, mobile, hub for each platform)
- Capacitor/Ionic localhost (for mobile apps)
- Development localhost (only in dev mode)

Preflight requests are handled correctly. Credentials are allowed only for whitelisted origins.

### 3.6 Security Headers

Content Security Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and XSS protection headers are set via Next.js middleware:

- `Content-Security-Policy`: Restricts script sources, prevents frame embedding (`frame-ancestors 'none'`)
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: no-referrer`

### 3.7 Audit Logging

Two-tier audit logging system:

**API-level** (`api/_lib/audit-logger.ts`): Fire-and-forget async logging with IP extraction, risk classification, convenience functions for auth events, data access, and data modification. HIPAA §164.308 compliant.

**Service-level** (`services/security/AuditLogger.ts`): Database-backed logging with compliance flagging (HIPAA_RELEVANT, GDPR_RELEVANT), risk level calculation, and action type classification (LOGIN_SUCCESS, PHI_VIEW, PHI_EXPORT, etc.).

Audit logs are stored in an **immutable table** protected by database triggers — they cannot be modified or deleted. Retention period: 7 years.

### 3.8 Security Database Infrastructure

8 security tables and 5 security views are defined in migration `010_create_security_infrastructure.sql`:

**Tables**: `audit_logs`, `security_incidents`, `encryption_status`, `compliance_checks`, `data_subject_requests`, `incident_response_team`, `security_policies`, `security_metrics`

**Views**: `v_recent_phi_access` (24h PHI access log), `v_mfa_enrollment` (MFA by role), `v_open_incidents` (open incidents by severity), `v_compliance_summary`, `v_data_subject_requests` (GDPR requests with SLA tracking)

### 3.9 Vulnerability Scanning

`VulnerabilityService.ts` provides a framework for dependency scanning (npm audit integration), compliance scanning (HIPAA, GDPR, SOC2, ISO 27001, NIST), CVE/CVSS scoring, and remediation tracking.

Tests use real assertions with mocked dependencies, cover edge cases, test async flows, and validate error scenarios. Test quality has been verified by reading representative samples.

**Coverage enforcement**: CI pipelines run `vitest --coverage` and block any PR that drops below threshold. Two tiers of threshold are enforced. A low global floor (statements ≥5%, branches ≥3%, functions ≥3%, lines ≥5%) acts as a regression tripwire across the frontend `src/**` tree; it is deliberately low because most of `src/` is UI covered by Playwright e2e rather than unit tests, and the serverless `api/` handlers are integration-tested (via the critical-path smoke tests) and excluded from the unit-coverage figure. On top of that, per-module thresholds gate the genuinely business-logic modules at their real coverage: the institutional data-service layer (`src/services/university/**` in core, `src/services/company/**` in pro) is held at roughly 65-90% and cannot regress. UI components are covered by Playwright e2e tests rather than unit tests.

**Configuration**: Vitest with jsdom environment, `@testing-library/jest-dom` matchers, `@vitest/coverage-v8` provider with text, json-summary, and lcov reporters.

## 9. Code Organisation

### 9.1 Current Structure

7 production repositories plus 4 shared package repositories, all with consistent internal structure:
- `api/` — Vercel serverless functions
- `api/_lib/` — Shared middleware (auth, CORS, DB, error handling, audit logging)
- `src/` — React application code
- `src/services/` — Business logic and external service integrations

- `database/migrations/` — Versioned SQL migrations
- `tests/` or `e2e/` — Test suites
- `.github/workflows/` — CI/CD pipelines

### 9.2 Code Sharing & Shared Package Architecture

The University and Pro platforms share approximately 50% of their code. Rather than a monorepo migration, the architecture uses **private NPM packages** hosted on GitHub to eliminate duplication while keeping repositories independent — a deliberate choice for M&A readiness (any repo can be transferred or sold independently).

Four shared packages are published and consumed by all production repositories:

| Package | Purpose | Consumers | Lines Consolidated |
|---------|---------|-----------|-------------------|
| `@mindmeasure/api-lib` | API middleware (auth, CORS, DB, audit, rate limiting, error handling) | core, pro-core | ~2,500 |
| `@mindmeasure/shared-services` | Security services (AuditLogger, MFA, VulnerabilityService, BackupRecovery), scoring engine, utilities | All 4 repos | ~5,300 |
| `@mindmeasure/shared-types` | TypeScript interfaces (security dashboard, multimodal pipeline, assessment, registration) | All 4 repos | ~920 |
| `@mindmeasure/shared-ui` | 48 shadcn/Radix UI components (button, card, dialog, table, sidebar, chart, toast, etc.) | core, pro-core | ~7,100 |

**Total: ~15,800 lines of duplicated code consolidated into shared packages.**

Each package uses **subpath exports** (e.g., `@mindmeasure/shared-ui/button`) so consumers only load the types and code they need, keeping build times fast. All packages are built with tsup (CJS + ESM + DTS).

Consuming repositories retain thin re-export wrappers (e.g., `export { Button } from '@mindmeasure/shared-ui/button'`) so all internal imports continue to work unchanged. This means:
- Zero disruption to developer workflow — imports like `@/components/ui/button` still work
- A single source of truth for shared code — fix a bug once, all repos get it on next `npm update`
- Each repo can still override or extend any component locally if needed

CI/CD pipelines (GitHub Actions) and Vercel deployments are configured with `GH_PAT` authentication to access these private packages during build.

## 10. Assessed Risk Areas

### 10.1 Vendor Dependency — ElevenLabs

The voice check-in feature (core product differentiator) depends on ElevenLabs Conversational AI. ~35 files touch this integration. A voice provider abstraction layer is planned on the roadmap to mitigate lock-in risk, with NVIDIA PersonaPlex identified as a potential self-hosted alternative at scale.

**Mitigation**: Provider abstraction is planned. The conversation data flow (transcripts, analysis) is platform-agnostic — only the real-time voice layer is ElevenLabs-specific.

### 10.2 Rate Limiting

Application-level rate limiting is implemented via sliding-window middleware (`api/_lib/rate-limiter.ts`) with four configurable tiers:

| Tier | Limit | Applied To |
|------|-------|-----------|
| `auth` | 8 req / 60s | All login, signup, MFA, and superuser code endpoints |
| `api` | 60 req / 60s | Database CRUD, reports, account deletion, session processing |
| `ai` | 12 req / 60s | Bedrock AI insights, Rekognition, SEO generation |
| `upload` | 10 req / 60s | S3 file uploads, media uploads |

Rate-limited responses return standard headers (`RateLimit-Limit`, `RateLimit-Remaining`, `Retry-After`) per IETF draft-ietf-httpapi-ratelimit-headers, with HTTP 429 and a `RATE_LIMIT_EXCEEDED` error code.

This sits on top of Vercel's built-in DDoS protection and database-level rate limiting for specific operations.

**Implementation caveat (stated for accuracy):** the middleware counter is an in-memory `Map` scoped to each warm serverless instance, with no shared store (no Redis). On Vercel this means the per-tier limits are enforced per warm instance rather than globally, and counters reset on cold start. It is an effective MVP control against casual abuse from a single source, and Vercel's edge DDoS protection covers volumetric attacks, but it is not a globally-coordinated limiter. A shared-store upgrade (Redis or Vercel KV) is the path to strict global enforcement if/when load justifies it.

### 10.3 Error Tracking

**Sentry** is integrated across all 4 active applications (both admin dashboards and both mobile apps):
- **Client-side**: `@sentry/react` with browser tracing, session replay on error (text masked for HIPAA), and `ErrorBoundary` wrapping at the app root
- **Server-side**: `@sentry/node` on both admin API backends, with 5xx errors automatically reported via the centralised error handler
- **Environment tagging**: Events are tagged by platform (university/pro) and app type (admin-dashboard/mobile/api) for filtering
- **Privacy**: Session replay masks all text and blocks all media. Sentry is disabled in localhost environments.

Additional error visibility comes from Vercel deployment logs, AWS CloudWatch/CloudTrail, the ecosystem health audit, and immutable audit logging.

### 10.4 Bus Factor

The platform has been built primarily by a single developer. This is mitigated by:
- Comprehensive documentation site (docs.mindmeasure.co.uk)
- Consistent patterns across all projects (same stack, same structure)
- 750+ automated tests with enforced coverage thresholds
- CI/CD pipelines that enforce quality standards
- TypeScript at maximum strictness (self-documenting code)
