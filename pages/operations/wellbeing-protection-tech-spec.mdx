## Phase 1a ‚Äî Database Foundations ‚úÖ BUILT

**Priority: High | Deployed | Migration: `run-migration-017`**

All subsequent phases depend on this. Additive schema changes with no breaking impact on existing code.

### What was built

**API file:** `mind-measure-core/api/admin/run-migration-017.ts`

Run via `POST /api/admin/run-migration-017` (superadmin auth required). Safe to call multiple times ‚Äî all statements use `IF NOT EXISTS`.

**Allowlist update:** `mind-measure-core/api/database/select.ts` ‚Äî added `disclosure_notifications` and `welfare_trigger_state` to `ALLOWED_TABLES`.

### Schema changes

```sql
-- Safeguarding contact fields on universities
ALTER TABLE universities
  ADD COLUMN IF NOT EXISTS safeguarding_contact_name  VARCHAR(255),
  ADD COLUMN IF NOT EXISTS safeguarding_contact_email VARCHAR(255),
  ADD COLUMN IF NOT EXISTS safeguarding_contact_phone VARCHAR(50),
  ADD COLUMN IF NOT EXISTS safeguarding_contact_role  VARCHAR(255);

-- Safeguarding consent fields on profiles
-- (Moved forward from Phase 5 ‚Äî simpler to run in one migration)
ALTER TABLE profiles
  ADD COLUMN IF NOT EXISTS safeguarding_consent_given_at  TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS safeguarding_consent_version   TEXT;

-- Disclosure notification log
CREATE TABLE IF NOT EXISTS disclosure_notifications (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id       UUID NOT NULL,
  university_id    UUID NOT NULL,
  trigger_type     TEXT NOT NULL,
  trigger_detail   JSONB NOT NULL DEFAULT '{}',
  score_data       JSONB NOT NULL DEFAULT '{}',
  recipient_name   TEXT,
  recipient_email  TEXT NOT NULL,
  sent_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  reviewed_at      TIMESTAMPTZ,
  reviewer_id      TEXT,
  review_outcome   TEXT CHECK (review_outcome IN ('confirmed', 'false_positive', 'additional_action')),
  review_notes     TEXT
);

CREATE INDEX IF NOT EXISTS idx_disclosure_notifications_student
  ON disclosure_notifications (student_id, sent_at DESC);

CREATE INDEX IF NOT EXISTS idx_disclosure_notifications_university
  ON disclosure_notifications (university_id, sent_at DESC);

CREATE INDEX IF NOT EXISTS idx_disclosure_notifications_pending_review
  ON disclosure_notifications (sent_at) WHERE reviewed_at IS NULL;

-- Welfare trigger state ‚Äî per-student rolling window counters
CREATE TABLE IF NOT EXISTS welfare_trigger_state (
  student_id                  UUID        PRIMARY KEY,
  consecutive_critical_weeks  INT         NOT NULL DEFAULT 0,
  crisis_flags_7day_window    INT         NOT NULL DEFAULT 0,
  crisis_flags_window_start   DATE,
  welfare_weeks_6wk_window    INT         NOT NULL DEFAULT 0,
  trailing_4wk_avg            NUMERIC(5,2),
  last_evaluated_week         DATE,
  updated_at                  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Verification:** Run `POST /api/admin/run-migration-017` from the superuser panel ‚Üí System tab. Confirm 200 OK. In Aurora, verify `\d universities` shows new columns, and both new tables exist.

## Phase 2 ‚Äî Compound Trigger Engine

**Priority: High | Blocked by: clinical threshold review**

New logic runs within the weekly reflection pipeline, after `runSafetyCheck()`. Evaluates all four compound triggers using per-student historical data.

### New function: `evaluateCompoundTriggers()`

**File:** `mind-measure-core/api/reflections/weekly.ts`

```typescript
interface CompoundTriggerResult {
  triggered: boolean;
  triggerType?: 'sustained_critical' | 'repeated_crisis_language' | 'score_collapse' | 'sustained_welfare_pattern';
  triggerDetail: Record<string, unknown>;
}

async function evaluateCompoundTriggers(
  client: Client,
  studentId: string,
  currentWeekAvg: number,
  currentWeekStart: Date,
  safetyCheckResult: SafetyCheckResult
): Promise<CompoundTriggerResult>
```

**Trigger A ‚Äî Sustained critical scores:**
Query `weekly_reflection_sends` for this student. Find the preceding week's average. If both `currentWeekAvg < 25` and `previousWeekAvg < 25` (with ‚â• 3 check-ins each week): return trigger A.

**Trigger B ‚Äî Repeated crisis language:**
Query `assessment_sessions` for `safety_flag = true` within the 7-day window ending today. If count ‚â• 2: return trigger B.

**Trigger C ‚Äî Score collapse:**
Query last 4 completed weeks from `weekly_reflection_sends` (excluding current week). Compute trailing average. If `currentWeekAvg ‚â§ trailingAvg - 40`: return trigger C.

**Trigger D ‚Äî Sustained welfare pattern:**
Query `weekly_reflection_sends` for `send_type = 'welfare'` in the 6-week rolling window ending today. If count ‚â• 3: return trigger D.

**State update:** After evaluation, upsert to `welfare_trigger_state` with latest values.

**Important:** A student already disclosed this week should not trigger a second disclosure. Check `disclosure_notifications` before firing.

## Phase 3 ‚Äî Automated Level 4 Notification Pipeline

**Priority: High | Blocked by: Phase 2, clinical review, DPA sign-off, SES verified identity**

When `evaluateCompoundTriggers()` returns `triggered: true`, call `sendSafeguardingDisclosure()`.

### New function: `sendSafeguardingDisclosure()`

**File:** `mind-measure-core/api/reflections/weekly.ts`

```typescript
async function sendSafeguardingDisclosure(
  client: Client,
  student: { id: string; name: string; email: string },
  university: { safeguarding_contact_name: string; safeguarding_contact_email: string },
  trigger: CompoundTriggerResult,
  scoreData: { currentAvg: number; trailingAvg: number; trend: string; lastCheckInDate: Date }
): Promise<void>
```

**Steps:**
1. Compose email body (plain text + HTML version) ‚Äî fields specified in ¬ß3.1 of the policy
2. Send via `SESClient.SendEmailCommand`:
   - `From`: `safeguarding@mindmeasure.co.uk` (new SES verified identity)
   - `To`: `university.safeguarding_contact_email`
   - Subject: `[Mind Measure] Student Welfare Concern ‚Äî Action Recommended` *(neutral ‚Äî no identifying info in subject)*
3. Log to `disclosure_notifications` table
4. Send Sentry alert: `Sentry.captureMessage('Level 4 disclosure sent', { level: 'warning', extra: { studentId, trigggerType } })`

**If `safeguarding_contact_email` is null:** Fall back to `universities.mind_measure_email`. Log the fallback. Alert Sentry that safeguarding contact fields are not populated.

**Idempotency:** Check `disclosure_notifications` for a disclosure sent for this student within the last 7 days before sending. If one exists, skip and log.

**Files (university first):**
- `mind-measure-mobile-final/src/components/SafeguardingConsentScreen.tsx`

**Screen behaviour:**
- Displays the onboarding copy (see ¬ß4.1 of the policy)
- Single button: "I understand ‚Äî continue"
- No opt-out path. The copy explains this directly.
- On confirmation: `PATCH /api/profiles/consent` ‚Äî writes `safeguarding_consent_given_at = NOW()` and `safeguarding_consent_version = '<T&C version string>'`
- Consent cannot be bypassed ‚Äî the check-in flow is blocked until this screen is completed

**State machine change (`MobileAppStructure.tsx`):**
- Add `'safeguardingConsent'` to the `OnboardingScreen` union type

- If null: route to `'safeguardingConsent'` screen
- Pro product consent screen is a **separate design track** ‚Äî different framing, different legal basis

## Phase 6 ‚Äî Per-Student Welfare Timeline + PDF Export

**Priority: High | No blocking dependency on clinical/legal ‚Äî can be built any time**

Purpose-built evidence retrieval tool. Chronological timeline of all safety-relevant events for a given student. Exportable as a structured PDF. Suitable for a coroner's inquiry, subject access request, or institutional review.

### API endpoint

**File:** `mind-measure-core/api/welfare-record/[studentId].ts`

```
GET  /api/welfare-record/:studentId          ‚Äî full welfare timeline
GET  /api/welfare-record/:studentId/export   ‚Äî PDF export
```

**Timeline events (all types merged, sorted chronologically):**

```typescript
type WelfareEvent =
  | { type: 'check_in';         date: string; score: number; safetyFlag: boolean }
  | { type: 'reflection_sent';  date: string; sendType: string; promptVersion: string }
```

**PDF export:** Server-side only (e.g. `pdfkit`) ‚Äî not client-side print. Date-ordered, all events with reasons.

## Phase 7 ‚Äî Real-Time Level 5 Crisis Response

**Priority: Critical | Blocked by: ElevenLabs webhook capability review, clinical advisor input**

The hardest and most architecturally complex piece. Requires a mechanism to detect crisis content during or immediately after a live Jodie check-in and fire a notification within minutes.

### Design options (to be evaluated)

**Option A ‚Äî Post-call webhook**  
ElevenLabs fires a webhook on conversation end. The webhook payload is received by `api/agents/crisis-webhook.ts`, which runs the conversation transcript through Bedrock for crisis classification. If crisis content detected: fire Level 5 notification.

Latency: minutes (acceptable if Jodie's in-session guidance is sufficiently strong).  
Risk: webhook support needs confirmation from ElevenLabs.

**Option B ‚Äî Check-in result re-analysis**  
The existing check-in result pipeline (which already runs Bedrock analysis) is extended to classify crisis content immediately and trigger a real-time alert.

Latency: seconds to minutes after conversation ends.  
Simpler to implement; relies on existing pipeline rather than new webhook.

**In-app response (both options):**  
The check-in result screen surfaces crisis resources immediately when crisis content is detected: Samaritans (116 123), university emergency contact, 999. These are shown before the score reveal.

**Level 5 notification format:**  
Same as Level 4, plus: indication that crisis-level content was detected during a live check-in; time of the check-in; `URGENT` flag in subject line; stronger recommendation: *"This student may be at immediate risk. We recommend urgent contact."*
