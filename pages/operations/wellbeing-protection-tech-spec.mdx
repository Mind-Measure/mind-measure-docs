## Phase 1a — Database Foundations ✅ BUILT

**Priority: High | Deployed | Migration: `run-migration-017`**

All subsequent phases depend on this. Additive schema changes with no breaking impact on existing code.

### What was built

**API file:** `mind-measure-core/api/admin/run-migration-017.ts`

Run via `POST /api/admin/run-migration-017` (superadmin auth required). Safe to call multiple times — all statements use `IF NOT EXISTS`.

**Allowlist update:** `mind-measure-core/api/database/select.ts` — added `disclosure_notifications` and `welfare_trigger_state` to `ALLOWED_TABLES`.

### Schema changes

```sql
-- Safeguarding contact fields on universities
ALTER TABLE universities
  ADD COLUMN IF NOT EXISTS safeguarding_contact_name  VARCHAR(255),
  ADD COLUMN IF NOT EXISTS safeguarding_contact_email VARCHAR(255),
  ADD COLUMN IF NOT EXISTS safeguarding_contact_phone VARCHAR(50),
  ADD COLUMN IF NOT EXISTS safeguarding_contact_role  VARCHAR(255);

-- Safeguarding consent fields on profiles
-- (Moved forward from Phase 5 — simpler to run in one migration)
ALTER TABLE profiles
  ADD COLUMN IF NOT EXISTS safeguarding_consent_given_at  TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS safeguarding_consent_version   TEXT;

-- Disclosure notification log
CREATE TABLE IF NOT EXISTS disclosure_notifications (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id       UUID NOT NULL,
  university_id    UUID NOT NULL,
  trigger_type     TEXT NOT NULL,
  trigger_detail   JSONB NOT NULL DEFAULT '{}',
  score_data       JSONB NOT NULL DEFAULT '{}',
  recipient_name   TEXT,
  recipient_email  TEXT NOT NULL,
  sent_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  reviewed_at      TIMESTAMPTZ,
  reviewer_id      TEXT,
  review_outcome   TEXT CHECK (review_outcome IN ('confirmed', 'false_positive', 'additional_action')),
  review_notes     TEXT
);

CREATE INDEX IF NOT EXISTS idx_disclosure_notifications_student
  ON disclosure_notifications (student_id, sent_at DESC);

CREATE INDEX IF NOT EXISTS idx_disclosure_notifications_university
  ON disclosure_notifications (university_id, sent_at DESC);

CREATE INDEX IF NOT EXISTS idx_disclosure_notifications_pending_review
  ON disclosure_notifications (sent_at) WHERE reviewed_at IS NULL;

-- Welfare trigger state — per-student rolling window counters
CREATE TABLE IF NOT EXISTS welfare_trigger_state (
  student_id                  UUID        PRIMARY KEY,
  consecutive_critical_weeks  INT         NOT NULL DEFAULT 0,
  crisis_flags_7day_window    INT         NOT NULL DEFAULT 0,
  crisis_flags_window_start   DATE,
  welfare_weeks_6wk_window    INT         NOT NULL DEFAULT 0,
  trailing_4wk_avg            NUMERIC(5,2),
  last_evaluated_week         DATE,
  updated_at                  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Verification:** Run `POST /api/admin/run-migration-017` from the superuser panel → System tab. Confirm 200 OK. In Aurora, verify `\d universities` shows new columns, and both new tables exist.

## Phase 1b — Safeguarding Contact Management ✅ BUILT

**Priority: High | Deployed | No prerequisites**

Allows superadmins to configure the named safeguarding contact at each university — the individual who will receive Level 4 automated disclosures.

### What was built

**API:** `mind-measure-core/api/universities/update-safeguarding.ts`

```
POST /api/universities/update-safeguarding
```

Body: `{ universityId, name, email, phone?, role? }` — Zod-validated. Updates four `safeguarding_contact_*` columns on the `universities` row.

**UI:** `mind-measure-core/src/components/superuser/SafeguardingContactPanel.tsx`

Inline expandable panel on each university card in the superuser panel → Universities tab. A `ShieldAlert` icon button is shown on every card:
- **Amber** — contact not yet configured (warning state)
- **Teal** — contact is configured

Clicking the button expands a four-field form (name, email, phone, role) that saves without navigating away. After saving, the card state updates in memory without a full reload.

**Important:** Until a university's safeguarding contact is configured, Level 4 automated notifications will fall back to `universities.mind_measure_email` — but this produces a Sentry alert flagging the misconfiguration.

## Phase 2 — Compound Trigger Engine

**Priority: High | Blocked by: clinical threshold review**

New logic runs within the weekly reflection pipeline, after `runSafetyCheck()`. Evaluates all four compound triggers using per-student historical data.

### New function: `evaluateCompoundTriggers()`

**File:** `mind-measure-core/api/reflections/weekly.ts`

```typescript
interface CompoundTriggerResult {
  triggered: boolean;
  triggerType?: 'sustained_critical' | 'repeated_crisis_language' | 'score_collapse' | 'sustained_welfare_pattern';
  triggerDetail: Record<string, unknown>;
}

async function evaluateCompoundTriggers(
  client: Client,
  studentId: string,
  currentWeekAvg: number,
  currentWeekStart: Date,
  safetyCheckResult: SafetyCheckResult
): Promise<CompoundTriggerResult>
```

**Trigger A — Sustained critical scores:**
Query `weekly_reflection_sends` for this student. Find the preceding week's average. If both `currentWeekAvg < 25` and `previousWeekAvg < 25` (with ≥ 3 check-ins each week): return trigger A.

**Trigger B — Repeated crisis language:**
Query `assessment_sessions` for `safety_flag = true` within the 7-day window ending today. If count ≥ 2: return trigger B.

**Trigger C — Score collapse:**
Query last 4 completed weeks from `weekly_reflection_sends` (excluding current week). Compute trailing average. If `currentWeekAvg ≤ trailingAvg - 40`: return trigger C.

**Trigger D — Sustained welfare pattern:**
Query `weekly_reflection_sends` for `send_type = 'welfare'` in the 6-week rolling window ending today. If count ≥ 3: return trigger D.

**State update:** After evaluation, upsert to `welfare_trigger_state` with latest values.

**Important:** A student already disclosed this week should not trigger a second disclosure. Check `disclosure_notifications` before firing.

## Phase 3 — Automated Level 4 Notification Pipeline

**Priority: High | Blocked by: Phase 2, clinical review, DPA sign-off, SES verified identity**

When `evaluateCompoundTriggers()` returns `triggered: true`, call `sendSafeguardingDisclosure()`.

### New function: `sendSafeguardingDisclosure()`

**File:** `mind-measure-core/api/reflections/weekly.ts`

```typescript
async function sendSafeguardingDisclosure(
  client: Client,
  student: { id: string; name: string; email: string },
  university: { safeguarding_contact_name: string; safeguarding_contact_email: string },
  trigger: CompoundTriggerResult,
  scoreData: { currentAvg: number; trailingAvg: number; trend: string; lastCheckInDate: Date }
): Promise<void>
```

**Steps:**
1. Compose email body (plain text + HTML version) — fields specified in §3.1 of the policy
2. Send via `SESClient.SendEmailCommand`:
   - `From`: `safeguarding@mindmeasure.co.uk` (new SES verified identity)
   - `To`: `university.safeguarding_contact_email`
   - Subject: `[Mind Measure] Student Welfare Concern — Action Recommended` *(neutral — no identifying info in subject)*
3. Log to `disclosure_notifications` table
4. Send Sentry alert: `Sentry.captureMessage('Level 4 disclosure sent', { level: 'warning', extra: { studentId, trigggerType } })`

**If `safeguarding_contact_email` is null:** Fall back to `universities.mind_measure_email`. Log the fallback. Alert Sentry that safeguarding contact fields are not populated.

**Idempotency:** Check `disclosure_notifications` for a disclosure sent for this student within the last 7 days before sending. If one exists, skip and log.

## Phase 6 — Per-Student Welfare Timeline + PDF Export ✅ BUILT

**Priority: High | Deployed | No blocking dependencies**

Purpose-built evidence retrieval tool. Chronological timeline of all safety-relevant events for a given student. Exportable as a structured PDF. Suitable for a coroner's inquiry, subject access request, or institutional review.

### What was built

**API:** `mind-measure-core/api/welfare-record/[studentId].ts`

```
GET /api/welfare-record/:studentId          — full welfare timeline (JSON)
GET /api/welfare-record/:studentId?format=pdf — PDF download
```

Merges three data sources into a single chronological event stream:

| Source | Event type | Fields |
|---|---|---|
| `fusion_outputs` | `check_in` | score, zone, risk level, safety flag, mood self-report, conversation summary |
| `weekly_reflection_sends` | `reflection_sent` / `reflection_held` | send type, hold reason, Bedrock flag, week start |
| `disclosure_notifications` | `disclosure` | trigger type, recipient, review outcome |

**PDF generation** uses PDFKit (server-side, pure Node.js). Output is A4, colour-coded by score zone, with red highlighting for safety events and disclosures. Includes safeguarding consent date and confidentiality footer. Suitable for coroner's inquiry, subject access request, or OIA complaint response.

**UI:** `mind-measure-core/src/components/superuser/WelfareTimeline.tsx`

Slide-in panel accessible from two entry points:
- "Timeline" button on each row in the Welfare Record → Trigger State table
- "View timeline →" link on each Pending Review card

Panel shows all events chronologically, check-in cards have expandable conversation summaries. "Export PDF" button in the header downloads the formatted document directly.

## Phase 7 — Real-Time Level 5 Crisis Response

**Priority: Critical | Blocked by: ElevenLabs webhook capability review, clinical advisor input**

> **Note — Pro product:** The Pro equivalent of Level 5 in-app response has been built and deployed as a separate track. See [Pro — Level 4 & 5 In-App Response](/operations/pro-level-4-5-spec). The university version below is architecturally more complex because it also involves notifying the institution's safeguarding contact.

The hardest and most architecturally complex piece for the university product. Requires a mechanism to detect crisis content during or immediately after a live Jodie check-in and fire a notification within minutes.

### Design options (to be evaluated)

**Option A — Post-call webhook**  
ElevenLabs fires a webhook on conversation end. The webhook payload is received by `api/agents/crisis-webhook.ts`, which runs the conversation transcript through Bedrock for crisis classification. If crisis content detected: fire Level 5 notification.

Latency: minutes (acceptable if Jodie's in-session guidance is sufficiently strong).  
Risk: webhook support needs confirmation from ElevenLabs.

**Option B — Check-in result re-analysis** ✅ Implemented in Pro  
The existing check-in result pipeline (which already runs Bedrock analysis) is extended to classify crisis content immediately. The Pro implementation uses `risk_level: high` from the fusion result to trigger a `CrisisInterstitial` full-screen component after the score reveal. No employer/institutional notification in the Pro version — in-app resources only.

For university: same pipeline extension, but additionally triggers a Level 5 notification to the safeguarding contact (same email mechanism as Level 4, with URGENT flag).

Latency: seconds to minutes after conversation ends.  
Simpler to implement; relies on existing pipeline rather than new webhook.

**In-app response (both options):**  
✅ Done for Pro. University equivalent to be built alongside the notification work. Crisis resources shown before returning to dashboard: Samaritans (116 123), university emergency contact, 999.

**Level 5 notification format (university only):**  
Same as Level 4, plus: indication that crisis-level content was detected during a live check-in; time of the check-in; `URGENT` flag in subject line; stronger recommendation: *"This student may be at immediate risk. We recommend urgent contact."*
