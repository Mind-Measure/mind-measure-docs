# Mind Measure System Architecture

Mind Measure turns a one to two minute conversation into a reliable read on how a person is doing. Behind that simple experience sits a cloud-native platform that records nothing the user has to fill in, analyses speech, voice and facial signal in parallel, fuses them into a single Wellbeing Score, and shows institutions only an anonymised, population-level picture. This page explains how that platform is put together, why the design is hard to replicate, and how it stays secure, compliant and inexpensive to run.

Everything runs on managed AWS services in the London region (eu-west-2), so data stays in the UK from end to end. The platform is serverless almost everywhere it can be, which means it scales with demand and costs close to nothing at idle.

## The shape of the system

Read the diagram top to bottom and it tells the whole story.

At the **client layer**, end users only ever meet native mobile apps: a student app and an employee app, both built from a shared React and TypeScript codebase and packaged with Capacitor for iOS and Android. Institutional staff use web dashboards (the University and Pro admin panels), and the Content and Conversational Insight hubs are separate Next.js applications. Students and employees never use a browser to access the platform.

At the **edge**, Vercel serves every web surface and hosts the serverless API, with TLS 1.3 terminated at the edge and a global CDN in front of static assets. Authentication is handled by AWS Cognito, which issues short-lived signed (RS256) JWTs that every API route verifies on each request.

The **check-in pipeline** is the heart of the product and the next section covers it in detail. Below it, the **data and storage layer** is Aurora Serverless v2 (PostgreSQL) for structured data and Amazon S3 for media, both encrypted. Wrapping everything are the **cross-cutting services**: key management, secrets, monitoring, audit and a web application firewall, all within the same UK region.

## The check-in pipeline: the core of the platform

Every user begins with a one-off baseline. Jodie, the conversational voice agent, walks them through two validated clinical screeners (the PHQ-2 for depression and the GAD-2 for anxiety) plus a self-reported mood rating and a short open conversation. That establishes the personal reference point against which every later check-in is measured.

From then on, a daily check-in is just a natural two minute conversation. There are no forms, sliders or multiple-choice questions. When the conversation ends, the platform runs three analyses in parallel:

- **Text analysis (70 per cent of the score)** is the primary, most clinically grounded signal. AWS Bedrock, running Claude 3.7 Sonnet, reads the conversation transcript and returns a structured result: a mood score, PHQ-2 and GAD-2 indicators, risk markers, sentiment and the salient themes the person raised.
- **Voice analysis (15 per cent)** extracts acoustic features (pitch, speaking rate, jitter, shimmer, pause patterns, energy) from the audio. These prosodic features are research-linked to mood and psychological state.
- **Facial analysis (15 per cent)** uses AWS Rekognition to read facial affect when the camera is on (smile frequency, eye contact, expression, tension).

A state machine orchestrates these three steps in parallel, then a **fusion engine** combines the modality scores into a single value from 0 to 100. The design is deliberately resilient. If a signal is missing, the weights redistribute automatically: with no camera, text takes 80 per cent and audio 20 per cent; with neither audio nor video, the score comes from text alone. A sanity floor stops a cautious audio or visual reading from dragging down a clearly positive conversation. The result is that the platform produces a meaningful score even on a poor connection in a noisy room.

The user sees their score, their trend and a summary of what they discussed. The institution sees none of it at the individual level, only aggregates.

## Why this architecture is hard to copy

The intellectual property is not any single component; it is the integration. Delivering a validated multimodal wellbeing score inside a two minute consumer conversation requires expertise in conversational AI, clinical assessment methodology, audio signal processing, computer vision and serverless backend engineering, all stitched into a mobile experience that people will actually complete every day. Each part is buildable on its own. The combination, calibrated and degrading gracefully across modalities, is the moat.

Two design choices reinforce it. First, the conversation data, transcripts, scores and analysis are entirely provider-agnostic: only the real-time voice layer depends on a specific vendor, which keeps switching costs low and protects against lock-in. Second, the privacy model (described below) is architectural, not a policy bolted on top, which is a trust signal that surveillance-style competitors cannot easily match.

## Data, storage and residency

Structured data lives in **Aurora Serverless v2** (PostgreSQL), encrypted at rest with AES-256 using KMS-managed keys, with automated backups and point-in-time recovery. The cluster auto-scales with load and scales down when quiet.

Media is held in **Amazon S3**, encrypted, with private access through signed URLs. Raw signal is treated as disposable: audio is processed and then deleted within 24 hours, and still images from facial analysis are deleted immediately after processing. The platform does not collect location, contacts, photos, other app usage or browsing history.

All data is encrypted at rest at the storage layer (Aurora storage encryption and S3, AES-256 with AWS KMS-managed keys) and in transit (TLS). Application-level, per-field encryption of the most sensitive columns (names, conversation transcripts, assessment results) is designed but **not yet enforced on the server-side write path**, so it should be treated as roadmap rather than in force: the back-office analysis pipeline currently reads transcripts and results in plaintext. See the [database reference](/architecture/database) for the current state.

The **separation principle**, that institutions receive only anonymised population-level data and never an individual's score, conversation or identity, is enforced in the data layer itself. The minimum cohort size of five is applied at the query level: the queries that would expose an individual simply do not exist.

## Security and compliance

Data in transit is protected with TLS 1.3 from the user to the edge, and TLS to the database behind it. Authentication runs on AWS Cognito with signed JWTs verified cryptographically on every request, multi-factor authentication available, and a separate, time-limited one-time-code flow for Mind Measure superusers. Access is role-based and scoped to a single institution, so an administrator can only ever see their own organisation's data.

API traffic passes through sliding-window rate limiting tuned per endpoint class (stricter on authentication and AI endpoints), on top of the platform's built-in DDoS protection. Sensitive-data operations are written to a tamper-evident audit log retained for seven years: this covers the generic admin data API (read, insert, update, delete), authentication events, user-management actions, and wellbeing report and AI-insight generation. The log is append-only and entries are not editable through the application.

The platform is built to meet the requirements of UK GDPR, HIPAA and SOC 2 Type II: the technical controls are enterprise-grade, and the remaining work is formal governance documentation and audit evidence for certification. The dedicated **Security Posture** document covers controls in full; the **Privacy and Legal** section covers data subject rights and lawful basis.

## Scale, performance and cost

Because the stack is serverless where it counts (Aurora Serverless v2, Lambda, Cognito, Bedrock, Rekognition), it scales automatically and incurs little to no cost at idle. Aurora flexes across a wide capacity range; the analysis functions scale out across concurrent check-ins without manual intervention.

The practical result is that running costs are under £1,000 per month at current usage. The main variable cost is the real-time voice provider, at roughly $0.10 per minute of conversation, which scales linearly with active users. A planned provider-abstraction layer (with a self-hosted option identified for higher volumes) addresses both the cost curve and vendor concentration as the platform grows.

## Reliability and observability

An automated ecosystem health audit runs every four hours, checking database connectivity and latency, API availability, authentication, configuration and voice-agent activity, and escalates by email when a critical issue or a sharp drop in health score appears. Sentry captures client and server errors across the applications, with session replay on error. CloudWatch and CloudTrail provide infrastructure and audit logging, retained for one year and seven years respectively.

For recovery, Aurora provides automated daily backups with point-in-time restore to a five minute granularity, and S3 holds media with cross-region durability. Target recovery objectives are under four hours to restore the database, under two hours for the application, and under six hours for full service, with a recovery point objective of under fifteen minutes for the database.

## Engineering practice

Every frontend shares one stack (React 19, TypeScript and Vite), with TypeScript configured at maximum strictness, including settings most production codebases leave off. Roughly 15,800 lines that were once duplicated across the University and Pro products have been consolidated into four private NPM packages (shared API middleware, services, types and UI), while the repositories themselves stay independent, which keeps each one transferable on its own and preserves M&A flexibility.

Quality is enforced rather than hoped for: 750-plus unit tests run in CI with coverage thresholds that block any pull request which lowers them, end-to-end tests run in Playwright (including visual regression on the baseline assessment), and pre-commit hooks enforce formatting and linting. Deployment is by push to the main branch, which triggers an automatic Vercel build, with preview deployments for every change. The **Technical Due Diligence Report** covers code quality and engineering practice in depth.

## Technology stack

### Frontend and mobile

| Layer | Technology |
|---|---|
| UI framework | React 19, TypeScript |
| Build tooling | Vite |
| Styling | Tailwind CSS, Radix UI |
| Mobile | Capacitor (native iOS and Android), live on the App Store and Google Play |
| Hubs | Next.js (Content and Conversational Insight) |

### Backend and data

| Layer | Technology |
|---|---|
| API | Vercel serverless functions (front door, most logic); AWS Lambda + API Gateway for the multimodal analysis/scoring engine. See [Backend](/architecture/backend) |
| Orchestration | State machine for parallel check-in analysis |
| Database | Amazon Aurora Serverless v2 (PostgreSQL) |
| Storage | Amazon S3 (encrypted, signed-URL access) |
| Authentication | AWS Cognito (RS256 JWT, MFA) |

### AI and analysis

| Signal | Service |
|---|---|
| Text (70 per cent) | AWS Bedrock, Claude 3.7 Sonnet |
| Voice (15 per cent) | On-device acoustic feature extraction |
| Facial (15 per cent) | AWS Rekognition |
| Conversation | ElevenLabs conversational voice AI (Jodie) |

### Platform and security

| Concern | Service |
|---|---|
| Region | AWS eu-west-2 (London), UK data residency |
| Encryption | AES-256 at rest (Aurora/S3 storage encryption, KMS), TLS in transit; application-level field encryption now in force for the verbatim transcript (`assessment_transcripts.transcript`, KMS envelope), with `fusion_outputs.analysis` field encryption a tracked follow-up |
| Secrets | AWS Secrets Manager |
| Monitoring | CloudWatch, CloudTrail, Sentry |
| Edge protection | Web application firewall, rate limiting, DDoS protection |
| Hosting and CI/CD | Vercel (auto-deploy from the main branch) |
