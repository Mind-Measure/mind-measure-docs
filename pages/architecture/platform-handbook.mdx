## Development model (current)

- **Local development is normal.** Run dev servers (`npm run dev`, etc.) to build and test features. The old “production-only / no localhost” rule in archived master docs is **obsolete**; do not propagate it.
- **Deploy to production** for most apps by merging to **`main`** and letting **Vercel** build (GitHub integration). Do **not** use `npx vercel --prod` as the default path; it is only for exceptions (see below).
- **Test before push.** Follow project verification (build, lint where applicable) before asking to deploy.
- **University / Pro parity:** If a change applies to both lines, mirror it unless the product owner says otherwise.

---

## Critical constraints

1. **Do not modify the check-in / assessment conversation pipeline** implemented with ElevenLabs and the multimodal assessment engine in production without an explicit, reviewed change plan. Treat it as frozen unless leadership approves a scoped change.
2. **Do not hardcode database credentials.** Use `getSecureDbConfig()` from `api/_lib/db-config.ts` (or the established pattern in each repo).
3. **Avoid partial commits** in any single repo: commit all related files together so Vercel never deploys half a change set.
4. **Deploy workflow:** push to `main` → Vercel auto-build. **Exceptions** (no GitHub auto-deploy on some projects): university hub, pro hub, and docs site may require manual Vercel deploy — confirm in the repo’s README or ops notes; still avoid casual `vercel --prod` outside that documented process.

## Stack (summary)

- **Frontends:** React, TypeScript, Vite, Tailwind, Radix (admin); motion on marketing sites.
- **Hubs / CMS / docs:** Next.js, PostgreSQL via `pg`, Tiptap in hubs.
- **APIs:** Vercel serverless; use `Client` + `await client.end()` in serverless handlers unless the repo standard says otherwise (e.g. Pool in long-lived Next server).
- **Data:** Aurora Serverless v2 (PostgreSQL), `eu-west-2`.
- **Auth:** Cognito for end users; superuser flows use email + one-time code patterns documented per app.
- **AI:** ElevenLabs (Jodie), Bedrock (e.g. Claude for text analysis), Rekognition where used; details in [Assessment](/assessment) and ops specs.

## Session protocol (agents and humans)

Before **writing code or plans** for work that affects production behaviour:

1. Read this **Platform handbook** (you are here).
2. Read the operations pack **in full** when reflections, welfare, or safeguarding apply:
   - [Hierarchy of help](/operations/hierarchy-of-help)
   - [Weekly reflection safeguards](/operations/weekly-reflection-safeguards)
   - [Weekly reflection](/operations/weekly-reflection)

3. If changing welfare or disclosure behaviour, also read [Wellbeing protection policy](/operations/wellbeing-protection-policy) and [Wellbeing protection tech spec](/operations/wellbeing-protection-tech-spec).
4. Read the **actual source files** you will change — documentation is not a substitute.
