# Database Schema Documentation

This document describes the MindMeasure PostgreSQL schema: tables, relationships and indexes.

import { Callout } from 'nextra/components'

<Callout type="warning">
**Platform note (read first).** Mind Measure runs on **AWS Aurora Serverless v2 (PostgreSQL), `eu-west-2`**, having migrated off Supabase (ADR 001). Two consequences for this page:

- **There is no Postgres row-level security (RLS) and no `auth.uid()` on Aurora.** Access control is enforced in the **application/API layer**: every data path goes through authenticated Vercel `/api/*` functions that verify a Cognito JWT (or the superuser one-time-code session) and scope queries by user/institution with parameterised SQL. The `CREATE POLICY ... auth.uid()` and `auth.users` trigger snippets below are **legacy Supabase artefacts kept for historical reference only**; they are not in force.
- **The live check-in result table is [`fusion_outputs`](#fusion_outputs).** The `assessment_sessions` and per-modality `text_analysis` / `audio_analysis` / `visual_analysis` tables describe an earlier design; `assessment_sessions` in particular stopped being written some months ago. They are retained here as transitional/legacy context, not the current write path.
</Callout>

## Schema Overview

The schema is organised around multimodal check-ins: a conversation transcript and per-modality signals are fused into a single wellness score stored in `fusion_outputs`.

## Table Definitions

### profiles
**Purpose**: Stores user profile information and preferences

```sql
CREATE TABLE public.profiles (
 id UUID NOT NULL DEFAULT gen_random_uuid() PRIMARY KEY,
 user_id UUID NOT NULL UNIQUE,
 display_name TEXT,
 first_name TEXT,
 last_name TEXT,
 avatar_url TEXT,
 wellness_goals JSONB DEFAULT '[]'::jsonb,
 assessment_frequency TEXT DEFAULT 'weekly'::text,
 onboarding_completed BOOLEAN DEFAULT false,
 created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
 updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

**Indexes**
```sql
CREATE UNIQUE INDEX idx_profiles_user_id ON profiles(user_id);
CREATE INDEX idx_profiles_created_at ON profiles(created_at);
```

**Triggers**
```sql
CREATE TRIGGER update_profiles_updated_at
 BEFORE UPDATE ON profiles
 FOR EACH ROW
 EXECUTE FUNCTION update_updated_at_column();
```

**Access control (Aurora):** there is no RLS on this table. The `/api/*` functions that read or write `profiles` verify the caller's Cognito JWT and scope every query to the authenticated `user_id`. (The Supabase `auth.uid()` policies that previously guarded this table no longer exist.)

### fusion_outputs
**Purpose**: Comprehensive wellness scores using fusion model with probabilities

```sql
CREATE TABLE public.fusion_outputs (
 session_id UUID NOT NULL PRIMARY KEY,
 user_id UUID NOT NULL,
 p_worse_fused DOUBLE PRECISION NOT NULL,
 p_worse_audio DOUBLE PRECISION,
 p_worse_visual DOUBLE PRECISION,
 p_worse_text DOUBLE PRECISION,
 p_worse_passive DOUBLE PRECISION,
 score INTEGER NOT NULL,
 score_smoothed DOUBLE PRECISION NOT NULL,
 uncertainty DOUBLE PRECISION NOT NULL,
 qc_overall TEXT NOT NULL DEFAULT 'reliable',
 drivers JSONB,
 model_version TEXT NOT NULL DEFAULT 'v1.0',
 created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

<Callout type="warning">**`model_version` caveat.** This column currently defaults to a static `'v1.0'` and does not record the actual model used. As the platform has swapped Bedrock text models over time (Claude 3 Haiku, Claude 3 Sonnet, Claude Haiku 4.5) and the conversation LLM (Claude Sonnet 4.5), scores are not reliably comparable across periods unless the real model id and scoring version are stamped per row. Hardening this is tracked as a follow-up (stamp `BEDROCK_TEXT_MODEL_LABEL` + scoring version into every `fusion_outputs` write).</Callout>

## Database Functions

### update_updated_at_column()
**Purpose**: Automatically updates the `updated_at` timestamp on record modification

```sql
CREATE OR REPLACE FUNCTION public.update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
 NEW.updated_at = now();
 RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### Profile creation (no `auth.users` trigger on Aurora)

On Supabase, profiles were created by a `handle_new_user()` trigger on `auth.users`. **That mechanism does not exist on Aurora** (there is no `auth.users` table). User identity lives in **AWS Cognito**; a `profiles` row is created/linked by the application during onboarding (via an authenticated `/api/*` call) keyed to the Cognito `sub`. The historical trigger is intentionally omitted here because reproducing it would be misleading.

## Security & Privacy

### Access control (application layer, not RLS)

Aurora does not use Postgres row-level security here. Instead:
- Every read/write goes through an authenticated Vercel `/api/*` function that verifies a Cognito JWT (or the superuser one-time-code session).
- Queries are scoped to the authenticated `user_id` / institution and use parameterised SQL (no string-built queries).
- There is **no direct database access from clients**; the mobile apps reach data only through these API endpoints.

### Data Encryption

**At Rest**
- Aurora storage encryption (AWS-managed keys) covers the database at rest, including automated backups and snapshots.
- Application-level field encryption (`PHIEncryption`) exists in the client codebase but is **not yet enforced server-side** on the API write path; treat server-side field-level PHI encryption as designed, not in force. (Tracked as a security follow-up.)

**In Transit**
- TLS for all database connections.
- **Certificate verification is being hardened.** A number of connection sites currently set `rejectUnauthorized: false` (accept-any-cert), which is *not* certificate pinning. Moving these to verify against the AWS RDS CA bundle is tracked as a security follow-up; until then, do not describe the DB channel as "pinned".

### Data Retention

**Automated Policies**
```sql
-- Example: Archive old sessions after 2 years
CREATE OR REPLACE FUNCTION archive_old_sessions()
RETURNS void AS $$
BEGIN
 -- Move to archive table
 INSERT INTO assessment_sessions_archive
 SELECT * FROM assessment_sessions
 WHERE created_at < NOW() - INTERVAL '2 years';
 
 -- Delete from main table
 DELETE FROM assessment_sessions
 WHERE created_at < NOW() - INTERVAL '2 years';
END;
$$ LANGUAGE plpgsql;
```

## Performance Optimization

### Indexing Strategy

**Primary Indexes**
- All foreign keys are indexed
- Date fields for temporal queries
- Score fields for analysis queries
- Composite indexes for common query patterns

**Query Optimization**
```sql
-- Optimised query for a user's recent check-in scores
EXPLAIN (ANALYZE, BUFFERS)
SELECT session_id, score, score_smoothed, uncertainty, qc_overall, created_at
FROM fusion_outputs
WHERE user_id = $1
ORDER BY created_at DESC
LIMIT 10;
```

### Partitioning

**Time-based Partitioning** for large tables
```sql
-- Partition assessment_sessions by month
CREATE TABLE assessment_sessions_y2024m01 PARTITION OF assessment_sessions
FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
```

## Backup & Recovery

### Backup Strategy
- **Point-in-time recovery**: 7-day retention
- **Daily full backups**: 30-day retention
- **Weekly archive backups**: 1-year retention
- **Annual compliance backups**: 7-year retention

### Recovery Procedures
```sql
-- Example: Restore specific user data
BEGIN;
-- Restore from backup timestamp
SELECT * FROM backup_restore_user_data('user_uuid', '2024-01-01 12:00:00');
COMMIT;
```
