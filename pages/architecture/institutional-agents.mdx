# Institutional Agents

Each university or company on the Mind Measure platform has its own ElevenLabs Jodie agent, personalised with institutional knowledge and optional custom questions. The agent feels embedded in the institution rather than generic.

**Status**: Live (Phases 1–3 complete)  
**Repositories**: `mind-measure-core`, `mind-measure-pro-core`, `mind-measure-mobile-final`, `mind-measure-pro-mobile`

---

## Architecture Overview

### Agent Hierarchy

```
Base Prompt (managed per product line)
  ├── University Agent: Worcester
  │     ├── Knowledge Base: compiled from CMS data
  │     └── Custom Questions: (none configured)
  ├── University Agent: LSE
  │     ├── Knowledge Base: compiled from CMS data
  │     └── Custom Questions: (none configured)
  ├── Company Agent: Simmonds Grabbit & Runne
  │     ├── Knowledge Base: compiled from CMS data
  │     └── Custom Questions: (none configured)
  └── Default Agent (fallback for unmatched institutions)
```

### Data Flow

```
CMS onboarding data → /api/agents/context → Plain-text knowledge document (URL)
                                                      ↓
Institution signup → /api/agents/create → ElevenLabs Clone API → New agent
                                                      ↓
                                            Knowledge Base URL registered
                                                      ↓
Mobile app → /api/agents/resolve → Returns elevenlabs_agent_id → Agent session
```

## API Endpoints

All endpoints live under `api/agents/` in both `mind-measure-core` and `mind-measure-pro-core`. Superuser authentication is required unless noted.

### Agent CRUD

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/agents/create` | POST | Clone a template agent via ElevenLabs, register knowledge base URL, insert database record |
| `/api/agents/list` | GET | List all institutional agents with institution details and question counts |
| `/api/agents/update` | PATCH | Update agent name or status |
| `/api/agents/delete` | POST | Delete ElevenLabs agent and KB document, remove database record |

### Questions

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/agents/questions` | GET | List questions for an agent (`?agentId=...`) |
| `/api/agents/questions` | POST | Create a new custom question |
| `/api/agents/questions` | PATCH | Update an existing question |

### Prompt Sync

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/agents/sync` | POST | Rebuild and push prompt + knowledge base for a single agent |
| `/api/agents/sync-all` | POST | Rebuild and push prompts for all active/draft agents |

### Configuration

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/agents/config` | GET | Retrieve all `agent_config` key/value pairs |
| `/api/agents/config` | PATCH | Update a config value (e.g. base prompt text) |
| `/api/agents/seed-prompt` | POST | Seed initial base prompt into `agent_config` |

### Public / Mobile

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/agents/resolve` | GET | Resolve `elevenlabs_agent_id` by institution ID. Returns `{ agentId, agentName, institutionType, fallback }`. No auth required. |
| `/api/agents/context` | GET | Generate plain-text knowledge document from CMS data for a given institution. No auth required (consumed by ElevenLabs). |

## Service Layer

### `InstitutionalAgentService.ts`

Located in `src/services/` in both core projects. Encapsulates all ElevenLabs API interactions.

#### ElevenLabs API Methods

| Method | ElevenLabs Endpoint | Purpose |
|--------|-------------------|---------|
| `cloneAgent(templateAgentId, name)` | `POST /v1/convai/agents/:id/duplicate` | Clone a template agent |
| `deleteAgent(agentId)` | `DELETE /v1/convai/agents/:id` | Delete an agent |
| `updateAgentPrompt(agentId, prompt)` | `PATCH /v1/convai/agents/:id` | Update the agent's system prompt |
| `addKnowledgeBaseUrl(agentId, url, name)` | `POST /v1/convai/knowledge-base` + `PATCH` | Create KB document from URL and attach to agent |
| `removeKnowledgeBaseDoc(docId)` | `DELETE /v1/convai/knowledge-base/:id` | Remove a KB document |

All calls use the `ELEVENLABS_API_KEY` environment variable.

#### `buildAgentPrompt()`

```typescript
function buildAgentPrompt(
  basePrompt: string,
  institutionName: string,
  institutionType: 'university' | 'company',
  questions: InstitutionalQuestion[]
): string
```

Constructs the full agent prompt by:

1. Starting with the product-specific base prompt (`base_prompt_university` or `base_prompt_company`)
2. Appending institutional context: *"You are speaking with students/employees from [name]..."*
3. Adding **required** questions as *"You must find a natural moment to ask..."*
4. Adding **optional** questions as *"Try to also explore..."*

## Base Prompt Management

The base operational prompt for Jodie is managed separately for each product line because the university and corporate products have different conversational requirements.

### How It Works

1. **Superuser UI**: The "Agents" tab in the Superuser Control Panel includes a collapsible "Base Agent Prompt" editor. In `mind-measure-core` it edits `base_prompt_university`; in `mind-measure-pro-core` it edits `base_prompt_company`.
2. **Storage**: Prompts are stored in the `agent_config` table as key/value pairs.
3. **Sync**: When a superuser clicks "Sync All Agents", the system:
   - Loads the appropriate base prompt for each agent's `institution_type`
   - Fetches active questions for each agent
   - Calls `buildAgentPrompt()` to construct the full prompt
   - Pushes the prompt to ElevenLabs via `updateAgentPrompt()`
   - Refreshes the knowledge base URL via `addKnowledgeBaseUrl()`

This allows the base Jodie prompt to be updated across all institutional agents from a single editor.

## Mobile App Integration

### Agent Resolution Flow

When a user starts a check-in, the mobile app resolves the correct agent:

1. **University app** (`CheckinAssessmentSDK.tsx`): If `user.university_id` exists, calls `/api/agents/resolve?university_id=:id`
2. **Corporate app** (`CheckinAssessmentSDK.tsx`): If `user.company_id` exists, calls `/api/agents/resolve?university_id=:id`
3. If the resolve endpoint returns a valid `agentId` (not fallback), that agent is used
4. Otherwise, falls back to the default template agent ID

The resolve endpoint queries `institutional_agents` for an active agent matching the institution ID:

```sql
SELECT elevenlabs_agent_id, agent_name, institution_type
FROM institutional_agents
WHERE institution_id = $1 AND status = 'active'
```

### Automatic Agent Creation

When a new university is created via the CMS (`createUniversity` in `src/features/cms/data.ts`), the system automatically calls `/api/agents/create` to clone an ElevenLabs agent and register the knowledge base URL.

## Knowledge Base

### Dynamic Context Document

The `/api/agents/context` endpoint generates a plain-text knowledge document from CMS data for a given institution. This document is hosted as a URL that ElevenLabs can index.

**Data sources** (already collected during CMS onboarding):
- Emergency and crisis resources
- Wellbeing centres and support services
- Locale and cultural context, and the internal welfare contact (routing only)
- Departments, divisions, and offices
- Mental health advocates per office (Pro), read from `companies.offices[].mentalHealthAdvocates`
- EAP and wellbeing programmes (Pro), read from `companies.settings.eap[]`
- Universities surface their own campus-based resources instead of the Pro EAP/advocate fields

**Pro EAP and advocates (added June 2026).** Companies record advocates against
each office (mirroring the office Site Lead) and Employee Assistance Programmes in
a dedicated **EAP** tab. EAP entries carry descriptive/narrative fields (tagline,
description, what they offer, how to access, cost) so `/api/agents/context` can
render them as conversational framing rather than a bare contact line, letting
Jodie say things like *"did you know your company subscribes to OpenUp..."* without
implying the employee pays. The `enabled` flag per programme gates whether it is
shared with the agent.

When CMS data is updated, the context URL serves the latest content. ElevenLabs re-indexes automatically.

### How Jodie Uses It

When a user mentions needing help or asks about services, Jodie references the knowledge base:
- *"Your university has a counselling service in the Student Life Centre — you can self-refer online."*
- *"Your company's EAP is through Health Assured — you can call them on 0800 028 0199, it's confidential."*

## Custom Questions — Conversational Insight

Institutions write natural-language questions that Jodie weaves into check-ins for the relevant audience. These are conversation prompts — not survey questions — and they're authored by the institution's own admins, not by Mind Measure staff. Full architecture in the [Conversational Insight v2 spec](/planning/conversational-insight-v2); the relevant facts for the agent layer are below.

### Question Properties

- **Category**: `localized_knowledge` (about the institution) or `strategic_insight` (about wellbeing topics). Admin-authored questions are always `strategic_insight`; `localized_knowledge` is reserved for MM-staff factual additions.
- **Priority**: `required` (Jodie should ask if the conversation allows) or `optional` (ask only if the conversation invites it naturally).
- **Status**: `draft` → `active` → `archived`. No unarchive path; clone-then-edit to re-run a theme.
- **Audience** (v2): JSONB cohort filter using the same `parseAudience` helper as Alerts. Defaults to `{ all: true }`. Persisted on `institutional_questions.audience`.
- **Time windows**: Optional `active_from` / `active_until` for seasonal questions.

### Authoring

- Company admins author via `admin.mindmeasurepro.com → Company → CMS → Questions`.
- University admins author via `admin.mindmeasure.co.uk → University → CMS → Questions`.
- MM superusers can also create questions via the legacy `/api/agents/questions` endpoint (broader access, e.g. to set `category: localized_knowledge`).

Every successful write fires a server-side `/api/agents/sync` for the institution's agent so the ElevenLabs prompt picks up the change within seconds. Sync failures are logged but don't fail the write — the row is in DB and the next manual sync (or the next scheduled cron) will reconcile.

### Library limits (v2)

- Soft warn at 50 currently-active questions per institution.
- Hard cap at 200 currently-active.
- Archived questions don't count toward either limit.

### Per-check-in exposure (v2, partial — schema only)

The resolver (`pickQuestionsForUser` in `api/_lib/questions.ts`) selects up to 2 audience-matching questions per check-in and atomically logs the ask in `institutional_question_asks`. Asked questions drop off that user's eligible pool whether they answered or stayed silent — silence is honoured.

**Note:** The per-session prompt override that calls this resolver is not yet wired into the mobile check-in path (Phase 4 in the [v2 spec](/planning/conversational-insight-v2)). Until that ships, the agent's stored prompt continues to include every active question and every user matching `{all:true}` hears all of them. v1 behaviour preserved as the fallback.

### Reporting

Shipped in Phase 3 of v1, enhanced in v2:

- Aggregate response counts per institution, surfaced via `ConversationalInsightCard` on the Overview.
- Bedrock-paraphrased samples and a 1-paragraph synthesis at n ≥ 8.
- K-anonymity floor of 5 responses gates display of sentiment, samples, and summary.
- Running stats per question (v2): Asked / Answered / Cohort.

### Future work

- CSV export of archived question aggregates (Phase 5 in v2 spec).
- Thread comparison view across runs of the same question (`thread_id` already in the schema).
- Per-user exposure stats on the library counters (depends on the Phase 4 resolver wiring being live).
